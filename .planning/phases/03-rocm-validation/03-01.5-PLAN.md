<objective>
Unblock Wave 2+ ROCm execution after the **D-31-A4.1 amendment** (substrate ordering retargeted from TensorWave-primary to **RunPod-primary**). Plans 03-02 through 03-06 currently import `orchestration.vultr_mi300x.provision` which raises `VultrProvisionError("UNSET sentinel")` on first call — none of those plans can run real spend until a primary MI300X provisioning module with the same public contract exists.

This plan supersedes the previous (TensorWave-targeted) version of 03-01.5 (committed ~10 minutes before this rewrite; never executed; preserved in git history). The decision pivot to RunPod is recorded in `.planning/phases/03-rocm-validation/03-01-AMENDMENTS.md` §D-31-A4.1. The driving empirical observations are:

1. RunPod's GraphQL `gpuTypes` query returns `AMD Instinct MI300X OAM` with `secureCloud: true`, `securePrice: 1.99` ($/GPU-hr), `maxGpuCount: 8`, `memoryInGb: 192`. The SKU is bookable per-GPU on the same Secure Cloud surface that hosts Phase 02's H100 work — same auth, same SDK, same `tools/probe_runpod_stock.py` poll loop.
2. RunPod's `lowestPrice` query currently returns `stock=None` across gpuCount={1,2,4,8} globally — the SKU is listed but momentarily out of stock. This matches the H100 pattern observed during Phase 02 and is resolved via the existing Phase-02 stock-poll pattern (poll-until-non-None with a watchdog).
3. RunPod tooling already in the repo from Phase 02 — `tools/probe_runpod_stock.py`, `tools/probe_runpod_dc.py`, `tools/find_runpod_volume.py`, `RUNPOD_API_KEY` in env, `results/_pulled/<pod-id>/` pull-back pattern, the `runpod` Python SDK in `requirements.lock` — covers ~80% of the provisioning + telemetry surface. The new module is mostly composition over existing primitives.
4. $0.28/GPU-hr cost premium over TensorWave ($1.99 vs $1.71). At Phase 0's planned ~23 MI300X GPU-hours, that is +$6.44 vs TensorWave — trivial against the $150 program ceiling.

This plan is MOSTLY OFFLINE/test-driven. The only real spend is the single budget-capped smoke pod (≤ $2.00 hard cap, 30 min watchdog) at Task 5 to prove the provisioning path end-to-end on a real MI300X instance. The smoke is opt-in: if Task 1's stock-poll reveals chronic stock=None (24h window) the plan HALTs at Task 1 with a clean re-plan trigger that re-activates the TensorWave investigation as 03-01.6.

**Why this plan exists (vs. the prior TensorWave-targeted 03-01.5):** Plan 03-01's Task 5 checkpoint surfaced amendment D-31-A4 (Vultr → TensorWave). A subsequent ~10-min empirical investigation surfaced amendment **D-31-A4.1**: RunPod publicly lists MI300X at $1.99/GPU-hr on Secure Cloud (was not present in CLAUDE.md §1.2 when CLAUDE.md was authored). Choosing RunPod gives Phase 0 a single substrate for both rails (CUDA H100 + ROCm MI300X), reuses ~80% of Phase 02's infrastructure, and eliminates the TensorWave provisioning-surface unknown that the prior 03-01.5 was scoped to investigate. The cost premium is trivial. CLAUDE.md §1.2's TensorWave-primary line is **deviated from intentionally** by D-31-A4.1; the deviation is documented in AMENDMENTS.md.

**Scope boundary:** This plan does NOT change `orchestration/vultr_mi300x.py` (sentinel UNSET is correct backup posture; tests still pass). It does NOT touch `orchestration/tensorwave_mi300x.py` (Phase-1 stub remains; only activated if HALT-STOCK fires). It does NOT rebuild `dockerfiles/rocm/Dockerfile` or re-pin `bench/images.lock.yaml` (the ROCm 7.12 base + the `rbox-pod-rocm:digest=pending` row from D-32-A1 are both correct as-is — `rbox-pod-rocm` derived-image push is still operator-deferred per 03-01-SUMMARY "Deferred Work #2"). It does NOT renumber existing plans 03-02..06 (the `03-01.5-PLAN.md` filename slots cleanly between 03-01 and 03-02 in lexical ordering — period > hyphen in ASCII).

**Bundling rationale (6 tasks acknowledged):** This plan exceeds the 2-3 task soft cap because it is the substrate-pivot enabler — it cannot meaningfully split. Task 1 (probe extension + stock-poll mechanic) feeds Task 5 (smoke). Tasks 2-4 (scaffold module + tests + dispatch shim) are tightly coupled and share a single AST-test fixture surface. Task 5 (smoke pod) cannot run until Tasks 1-4 commit. Task 6 (operator-verify) cannot run until Task 5's smoke produces evidence. Splitting would push the smoke pod into a separate plan with its own checkpoint thrash, doubling planning overhead for a $2 spend. Plan-budget threshold is accepted as-is on the same grounds Plan 03-01 used (4 auto + 1 checkpoint).

**Plan-level pass criteria** (three branches; Task 1 + Task 5 outputs select one):
- **PROCEED-RUNPOD** — Task 1 polls MI300X stock and observes ≥1 non-None lowestPrice offer within the polling window; Task 5 smoke pod creates+boots+terminates under the $2 cap; rocminfo returns a gfx94X line. Wave 2 (Plan 03-02) is unblocked; downstream plans import `orchestration.mi300x.provision` with `RBOX_MI300X_PROVIDER=runpod` (default) and execute against real MI300X without further re-planning.
- **HALT-STOCK** — Stock-poll returns None across 24h of polling at the cadence documented in `docs/runpod_mi300x_runbook.md` (e.g., every 5 min). Plan halts. Re-activate TensorWave provisioning investigation as a new 03-01.6 plan (original scope of the superseded 03-01.5 TensorWave version) — orchestration/tensorwave_mi300x.py + provisioning-surface research. Do NOT proceed to Wave 2 on RunPod under HALT-STOCK.
- **HALT-COST** — Task 5 smoke spend approaches the $2 cap or pod fails to reach RUNNING within 30 min watchdog. Escalate; do NOT proceed to Wave 2. Operator decides between (a) re-attempt with higher cap, (b) HALT-STOCK fallback, (c) HALT-AND-DOWNGRADE Phase 0 to CUDA-only per DR-31.
