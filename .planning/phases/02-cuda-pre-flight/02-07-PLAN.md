# Phase 02 Plan 07: Pod Multi-Service Startup + Corpus Bundling + Operator Rsync Setup

Closes the smoke-readiness gap surfaced 2026-05-07 while preparing to fire DEV-1018 (P2.1 smoke). Plan 02-06 baked `pod_entrypoint.sh` into a custom image and proved the bootstrap path end-to-end. The smoke path was untested and turns out to depend on three preconditions that 02-04 implicitly assumed but never wired: in-pod service orchestration, corpus availability, and operator-side rsync receiver.

## Discovery (2026-05-07, post-02-06 audit)

After 02-06 shipped (DEV-1035 Delivered), the operator asked "confirm the smoke plan is set up correctly, then smoke test." The audit surfaced three blocking gaps before any real-spend smoke could be attempted:

### Blocker 1 — In-pod multi-service orchestration missing

`gates/g1/runner.py` defaults:
```
--vllm-url http://127.0.0.1:8000
--chatterbox-url http://127.0.0.1:8004
--kokoro-url http://127.0.0.1:8005
--whisper-dir /models/distil_whisper_large_v3_int8
```

`tools/pod_entrypoint.sh` smoke/sanity branch:
```bash
if [[ "$GATE" == "smoke" ]]; then
    python -m gates.g1.runner --gate=smoke --n-calls=5 --corpus=corpus_500 &
fi
```

No `vllm serve`, no Chatterbox FastAPI server, no Kokoro FastAPI server, no health checks. The base `vllm/vllm-openai:v0.10.0` image's CMD did start vLLM, but 02-06 overrode ENTRYPOINT to point at `pod_entrypoint.sh` — losing that startup. Result: runner would `Connection refused` on the first STT call.

### Blocker 2 — Corpus audio not in the image

`.dockerignore` from 02-06 excludes `assets/corpus_500/`, `assets/corpus_g711/`, `assets/corpus_hesitation/` (5.7 GB combined). For smoke, only `corpus_500` is referenced (manifest rows tagged `corpus: corpus_500`). Bundling the smoke corpus is a small image-size hit (~150 MB compressed) and removes a runtime synthesis dependency.

### Blocker 3 — Operator-side rsync receiver not configured

`OPERATOR_HOST`, `SSH_PUBKEY`, `SSH_PRIVATE_KEY` are all UNSET in the operator's shell. `pod_entrypoint.sh:_setup_ssh` no-ops without `SSH_PRIVATE_KEY`; `_shutdown` skips `tools/rsync_results.sh` without `OPERATOR_HOST + ~/.ssh/id_ed25519`. Result data would be produced on the pod but never makes it back to `~/RBOX/results/smoke/`. Driver's `_validate_smoke` then fails sub-criterion (a) `5_rows` with `error: "no JSONL found"`.

## Tasks

### T1 — Bundle Chatterbox-TTS server into the image

**Files modified:** `Dockerfile`, `pyproject.toml` (cuda extras)

Add to the cuda-extras pip install in the Dockerfile (and to `pyproject.toml [project.optional-dependencies].cuda` for symmetry):

```
chatterbox-tts>=0.1   # Resemble AI client lib + model loader; see CLAUDE.md §5.1
```

Plus `git clone` or `pip install` the `devnen/Chatterbox-TTS-Server` FastAPI wrapper into `/opt/chatterbox-server`. CLAUDE.md §5.1 calls out this is the only actively-maintained ROCm fork; for the CUDA path on H100 we use the upstream main branch which has matching CUDA support. The server takes a model path (we'll point at `/models/ResembleAI__chatterbox/<rev>/`).

**Acceptance:** image build succeeds. `docker run --rm rbox-pod:t1-test python -c "import chatterbox; print(chatterbox.__version__)"` exits 0.

### T2 — Bundle Kokoro-FastAPI server into the image

**Files modified:** `Dockerfile`

Add `git clone https://github.com/remsky/Kokoro-FastAPI.git /opt/kokoro-server` (the upstream non-ROCm fork is what we want for CUDA H100 — the `moritzchow/Kokoro-FastAPI-ROCm` fork referenced in CLAUDE.md §5.2 is for Phase 03 MI300X work). `pip install -r /opt/kokoro-server/requirements.txt`. Configure to load model from `/models/hexgrad__Kokoro-82M/<rev>/`.

**Acceptance:** image build succeeds. Server can be invoked with `python -m kokoro_fastapi --port 8005 --model-dir /models/hexgrad__Kokoro-82M/<rev>` and responds 200 on `/health` within 60 sec.

### T3 — Multi-service startup orchestration in `pod_entrypoint.sh`

**Files modified:** `tools/pod_entrypoint.sh`

Add a new function `_start_inference_services()` that:

1. Resolves the model paths from `/models/.bootstrap_index.json` (cache_bootstrap's index) — never hardcode the SHA, always read it.
2. Starts vLLM in background:
   ```bash
   python -m vllm.entrypoints.openai.api_server \
       --model "$QWEN_DIR" \
       --quantization awq \
       --port 8000 \
       --guided-decoding-backend xgrammar \
       --max-model-len 4096 &
   VLLM_PID=$!
   ```
3. Starts Chatterbox-TTS-Server on port 8004.
4. Starts Kokoro-FastAPI on port 8005.
5. Health-checks all three with a 90-sec budget per service (curl `/health` or `/v1/models`); if any fails to become healthy, log error and `exit 1` so the pod tears down rather than wasting time on a half-broken stack.
6. Records the three PIDs so `_shutdown` can SIGTERM them gracefully before the audit step.

The whisper directory (`--whisper-dir /models/distil_whisper_large_v3_int8`) is read by faster-whisper directly inside the gate-runner process — no separate server needed.

Call sequence: `_setup_ssh` → `_start_cost_watch` → `_start_watchdog` → `_start_inference_services` → gate-runner exec.

**Acceptance:** local docker integration test (one-off, manually verified): `docker run --rm --gpus all -e GATE=smoke -e BOOTSTRAP_MODE=0 rbox-pod:t3-test bash /workspace/tools/pod_entrypoint.sh` boots all three servers, prints `[entrypoint] all services healthy`, then either runs the gate or hits the watchdog. (Smoke gate will still fail without corpus / rsync until T4/T5 land — that's fine.)

### T5 — Operator rsync receiver setup + checklist update

**Files modified:** `docs/OPERATOR-CHECKLIST-PHASE-02.md`

The operator's workstation (`bob@bob-TB250-BTC`) needs:

1. **Reachable host**: Tailscale (preferred — no public IP exposure) OR a public IP with port 22 forwarded.
2. **SSH server running** (`sudo systemctl status ssh` — install `openssh-server` if needed).
3. **Ed25519 keypair scoped to RBOX**: `ssh-keygen -t ed25519 -f ~/.ssh/id_rbox_pod -N ""` (no passphrase — it's a single-purpose key for benchmark-result rsync).
4. **`authorized_keys` entry**: append `~/.ssh/id_rbox_pod.pub` to `~/.ssh/authorized_keys` with a `from="<runpod-egress-cidrs>"` restriction if the operator has public IP exposed; not needed for Tailscale.
5. **Env exports** before `--mode smoke`:
   ```bash
   export OPERATOR_HOST="100.x.y.z"   # Tailscale IP, or public hostname
   export SSH_PUBKEY="$(cat ~/.ssh/id_rbox_pod.pub)"
   export SSH_PRIVATE_KEY="$(cat ~/.ssh/id_rbox_pod)"
   ```
   (`tools/run_preflight.py` reads these and forwards to `provision()` which forwards to pod env.)

Update `docs/OPERATOR-CHECKLIST-PHASE-02.md` §4 ("Real-spend smoke") to add this sub-section as a hard pre-condition before `--mode smoke` is run.

**Acceptance:** operator runs the checklist commands; `ssh -i ~/.ssh/id_rbox_pod operator@$OPERATOR_HOST "echo ok"` succeeds from another host (proxy test); env exports are non-empty.
