# Phase 02 Plan 06: Custom Pod Image with Baked-in Entrypoint

## Incident Summary (2026-05-06)

Operator ran `uv run python -m tools.run_preflight --mode bootstrap` (Plan 02-05 deliverable, post-merge). Pod `zkqbit98s0uulf` provisioned cleanly on H100 SXM ($2.99/hr) with `BOOTSTRAP_MODE=1` env var. Twenty minutes later the driver was still polling `status=RUNNING` with `cumulative=$0.00` — no progress, no exit. Operator killed the pod via the runpod SDK (`runpod.terminate_pod("zkqbit98s0uulf")`).

### Root cause

`orchestration/runpod_h100.py:provision()` calls `runpod.create_pod(image_name="vllm/vllm-openai:v0.10.0", env={...,"BOOTSTRAP_MODE":"1",...})`. The RunPod SDK does not override the image's container CMD/ENTRYPOINT unless `docker_args` is passed. The upstream vLLM image's CMD is the OpenAI server. So the container booted vLLM trying to serve a model it didn't have, sat in a startup loop, and ignored every harness env var. `tools/pod_entrypoint.sh` is in the operator-side repo at `/home/bob/RBOX/tools/pod_entrypoint.sh` — **it was never copied into the image and was never invoked**.

The Plan 02-05 SUMMARY's claim "the pod entrypoint reads BOOTSTRAP_MODE=1 and short-circuits to cache_bootstrap" was unverified — the E2E test mocked the RunPod SDK and asserted `BOOTSTRAP_MODE` made it into the kwargs, but never validated that the kwargs would actually run the entrypoint on a real pod. Process gap: Plan 02-05's E2E test asserted env-var injection without asserting entrypoint reachability.

### Cost impact

Pod ran ~21 min × $2.99/hr ≈ $1.05 wasted. Within the $0.67 ledger authorization plus 02-05's $17 contingency. No downstream cost-cap breach. Cost-watch's `cumulative=$0.00` reading is a separate (small) issue: the adapter's first sample lands before billing starts to accrue; the actual final spend would have been ~$1.05.

## Fix

A single-image baked-in-entrypoint approach replaces the bare-upstream image. Subsequent gate pods use the same image; `BOOTSTRAP_MODE`/`GATE` env vars are now read by the image's `ENTRYPOINT` (= `tools/pod_entrypoint.sh`).

### Files added

- `Dockerfile` — `FROM vllm/vllm-openai:v0.10.0`, installs OS deps (`rsync`, `openssh-client`, `ca-certificates`), best-effort installs `runpodctl`, pip-installs `requirements.lock` plus the [cuda] extras (faster-whisper, livekit-agents/plugins, xgrammar, httpx[http2]), copies the harness, sets `ENTRYPOINT ["bash", "/workspace/tools/pod_entrypoint.sh"]`.
- `.dockerignore` — excludes `assets/corpus_*` (5.7 GB regenerable from seeds), `results/`, `secrets/`, `.git`, `.planning/`, `tests/`, `docs/`, Python cache dirs.
- `scripts/build_pod_image.sh` — `docker buildx build --platform linux/amd64`; emits the immutable `@sha256:…` digest after a `--push` so the operator can paste it into `_DEFAULT_IMAGE`.

### Files modified

- `orchestration/runpod_h100.py` — `_DEFAULT_IMAGE` set to a sentinel (`"rbox/pod:UNSET-run-scripts-build_pod_image.sh-and-pin-digest"`) that fails RunPod's image-pull check loudly. Comment explains the incident and the build → push → pin workflow. CLAUDE.md §2.3 demands digest pinning, not tag pinning.

## Operator Tasks (sequential, gated)

### Task 1 — Build the image locally

```bash

```

Expected: `linux/amd64` build succeeds, vllm/vllm-openai layer pulled (~10 GB compressed), final image ~12-14 GB compressed. Verify with `docker images rbox-pod`.

### Task 2 — Push and resolve the digest

```bash

```

The script prints the digest-pinned reference: `docker.io/<your-handle>/rbox-pod@sha256:abcdef...`. Copy this string.

### Task 3 — Pin the digest in `_DEFAULT_IMAGE`

Replace the sentinel in `orchestration/runpod_h100.py:32` with the pasted digest reference. Commit:


## Test Plan

Unit/E2E: no new tests required. The existing 02-05 E2E (`tests/test_run_preflight_e2e.py`) still covers the env-var injection contract. The reachability gap is a runtime-only concern; the only honest test for "the entrypoint actually runs" is the operator's real-spend re-run (T4). Adding a Docker-level integration test (build + smoke) is deferred to Phase 4 if recurrence justifies it.

## Out of Scope

- ROCm sibling image (`Dockerfile.rocm`) for Phase 03 MI300X path. Tracked in Phase 03 prep, not here.
- Slimmer bootstrap-only image (`python:3.11-slim` base, no vllm/torch). Optimization that would shave ~10 GB off the bootstrap pull but adds image-management complexity. Defer until cost or build-time pain justifies.
- runpodctl in-pod self-stop testing. Best-effort install is in the Dockerfile; pod_entrypoint.sh tolerates absence. Verify behavior in Phase 03 if/when it bites.

## Process Note

Plan 02-05's E2E test verified env-var injection but not entrypoint reachability — those are different contracts and conflating them caused this incident. Going forward, any plan that injects env vars to be read by an image-side script must include either (a) an integration test that boots the actual image, or (b) an explicit acceptance criterion stating "operator real-spend run validates entrypoint executes." Adding to the project's verification-style heuristics carry-forward.
