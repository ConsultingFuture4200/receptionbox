# Phase 02 Plan 08: image_digest + git_commit lineage capture (DEV-1021)

**Note: this plan is recorded retroactively** to capture commit `34c3607` (`fix(02-08): populate image_digest + git_commit in result rows (DEV-1021)`), which landed without a PLAN.md file. The commit message is detailed enough that the fix is fully traceable; this file exists so the phase manifest is internally consistent (every shipped change has a plan record) and so `/gsd-audit-uat` does not flag REPRO-03 as having partial data lineage.

## Discovery

Plan 02-07 closed PREFLIGHT-01 with smoke verdict `pass=True` (run `2f6b…`, session `20260509T231720Z`). Inspecting the result rows surfaced two REPRO-03 provenance fields landing as literal placeholders:

- `image_digest = "pending"` — `substrate/cuda.py:_lookup_image_digest()` was reading `bench/images.lock.yaml` for the BASE `vllm/vllm-openai` entry, whose digest field is intentionally `pending` (we deploy the derived `rbox-pod` image, not the base).
- `git_commit = "unknown"` — `gates/_runner_base.py:_git_commit()` was calling `git rev-parse HEAD`, but the pod has no `.git` directory (excluded by `.dockerignore`).

REPRO-03 was marked `[x]` in `REQUIREMENTS.md` on the basis of schema enforcement (the pydantic `GateResult` validation rejects missing fields). The data was placeholder-populated. Schema-vs-data is the same trap that 02-05 flagged for REPRO-02.

## Tasks (as shipped in commit 34c3607)

### T1 — Forward deployed image_ref via pod env

**File modified:** `orchestration/runpod_h100.py`

`provision()` now forwards `RBOX_IMAGE_DIGEST=image_ref` in the pod's environment block. The pod-side substrate reads the env var first; lockfile path preserved as fallback for substrates that don't go through `provision()` (e.g., local dev, future ROCm cohort).

### T2 — Pod-side image_digest read prefers env var

**File modified:** `substrate/cuda.py`

`_lookup_image_digest()` now:
1. Reads `RBOX_IMAGE_DIGEST` env var first.
2. Accepts either bare `sha256:…` or full `registry/repo@sha256:…` and parses out the digest.
3. Falls back to lockfile read (Phase 1 path) when env var is unset.

### T3 — Bake operator-side git commit into the image

**Files modified:** `Dockerfile`, `scripts/build_pod_image.sh`

`scripts/build_pod_image.sh` now resolves `GIT_COMMIT=$(git rev-parse HEAD 2>/dev/null || echo unknown)` and passes it as a Docker build arg. `Dockerfile` declares `ARG GIT_COMMIT=unknown` **after** the heavy `COPY . /workspace/` and pip layers (so a HEAD change doesn't invalidate ~16 GB of cached layers), then `RUN echo "$GIT_COMMIT" > /workspace/.git_commit`.

### T4 — Pod-side git_commit read prefers baked file

**File modified:** `gates/_runner_base.py`

`_git_commit()` fallback chain is now: `git rev-parse HEAD` → read `/workspace/.git_commit` → `"unknown"`. The operator workstation hits the subprocess path (real `.git`); the pod hits the baked-file path.

### T5 — Build + push image v18, re-pin digest

`scripts/build_pod_image.sh ghcr.io/consultingfuture4200/rbox-pod:v18 --push` baked the new commit into `/workspace/.git_commit`. `_DEFAULT_IMAGE` in `orchestration/runpod_h100.py` updated from the v16 digest (`sha256:3244985d…`) to v18 (`sha256:abcf19f8…ea9d217`).

### T6 — Verify on real spend

Single-gate G2 diag pod (`jow8x9kugpkgxm`, session `20260510T132812Z-dev1021-verify.json`) produced result rows with `image_digest=sha256:abcf19f8d84c…` and `git_commit=f049bb87…` matching the deployed image and HEAD at v18 build time. WER also re-confirmed at 2.55% (DEV-1083 fix intact).
